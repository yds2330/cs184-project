A custom Iris/Optifine shader pack built from scratch: deferred shadow mapping, screen-space reflections, and animated water — running live in Minecraft Java Edition.

00 — Abstract

Abstract

We built a custom Minecraft Java Edition shader pack using the Iris/Optifine shader API, transforming a vanilla, flat-lit scene into one with directional sun shadows, screen-space reflections on reflective materials (glass, ice, metal and gem blocks, and water), and animated water whose surface normals are driven by a procedural sum-of-sines height field. Our goal was to apply core real-time graphics concepts from CS184 — deferred rendering, shadow mapping, screen-space ray marching, and the relationship between height fields and surface normals — to a context that millions of people engage with daily. Starting from near-empty stub shaders (gbuffer passes that output albedo × lightmap, a shadow pass that writes constant white, and a composite pass that does nothing), we built every shading effect ourselves across eight GLSL shader files. The result runs at interactive frame rates on consumer GPU hardware inside Minecraft Java Edition.
01 — Technical Approach

Technical Approach

Iris/Optifine enforces a deferred rendering pipeline structure — fixed stage names, fixed texture slots, fixed uniform names — but every line of shading logic inside each stage is authored entirely by us. The pipeline as we built it:

gbuffers_terrain.vsh/fsh  ─┐
                            ├──▶  colortex0   // albedo × lightmap  (lit scene color)
gbuffers_water.vsh/fsh    ─┤
                            └──▶  colortex1   // view-space normal (rgb) + reflectivity (a)  ← added by us
                                      │
                                      ▼
                            composite.vsh/fsh  ──▶  final.vsh/fsh  ──▶  screen
                                      ▲
              shadow.vsh/fsh  ──▶  shadowtex0   // 1024×1024 depth from the sun's POV
Feature 01

Directional Shadow Mapping

Sun shadows via a 1024×1024 orthographic shadow map reprojected through the depth buffer. Occluded pixels darkened 60%.

Feature 02

Material Tagging

Per-block reflectivity assigned at the vertex stage using Iris's block.properties + mc_Entity.x.

Feature 03

Screen-Space Reflections

32-step view-space ray march with Schlick Fresnel modulation. Runs only on pixels marked reflective.

Feature 04

Animated Water

Two-wave sum-of-sines height field, gradient-derived normals via finite differences, anchored in world space.

1 · Directional Sun Shadows

The shadow pass is configured in shaders.properties with shadow.resolution=1024. shadow.vsh transforms geometry using ftransform() — applying shadowProjection × shadowModelView, an orthographic projection from the sun's point of view. shadow.fsh outputs gl_FragColor = vec4(1.0), letting the driver write depth into shadowtex0.

In composite.fsh, we reconstruct each pixel's world position from the depth buffer via inverse projection matrices, re-project it into the sun's clip space, and compare depths. A bias of 0.005 prevents shadow acne. Occluded pixels are darkened by 60%:

vec4 ndcPos        = vec4(uv * 2.0 - 1.0, depth * 2.0 - 1.0, 1.0);
vec4 viewPos       = gbufferProjectionInverse * ndcPos;
vec4 worldPos      = gbufferModelViewInverse * vec4(viewPos.xyz / viewPos.w, 1.0);
vec4 shadowClip    = shadowProjection * (shadowModelView * worldPos);
vec3 shadowUV      = shadowClip.xyz / shadowClip.w * 0.5 + 0.5;
float inShadow     = (shadowUV.z - 0.005 > texture2D(shadow, shadowUV.xy).r) ? 1.0 : 0.0;
float shadowFactor = 1.0 - inShadow * 0.6;

2 · Material Tagging via block.properties and mc_Entity

Vanilla Minecraft's geometry stream carries no material identity. Iris's block.properties maps block names to integer IDs exposed per-vertex as mc_Entity.x. We declare three classes: ID 100 (glass, ice — reflectivity 0.7), ID 101 (iron, gold, diamond, emerald, netherite blocks — reflectivity 0.5), and ID 102 (water — reflectivity 0.6). These values are assigned in the vertex shader and packed into colortex1.a via the fragment shader.

3 · Augmented G-Buffer (colortex1)

Both gbuffer stages declare /* DRAWBUFFERS:01 */, enabling a second render target. colortex1 packs the view-space normal (remapped to [0,1] via * 0.5 + 0.5) and reflectivity into a single RGBA8 texture. The view-space normal is normalize(gl_NormalMatrix * gl_Normal) — the inverse-transpose of the modelview upper 3×3, correct for direction vectors under arbitrary linear deformations.

4 · Screen-Space Reflections (SSR)

For every pixel with reflectivity > 0.01, we reflect the view direction about the surface normal (R = V − 2(V·N)N) and march 32 steps of 0.3 view-space units. At each step we project back to screen UV, sample the depth buffer, and check whether the ray has passed behind visible geometry — indicated by diff = rayPos.z − scenePos.z; diff < 0.0 && diff > −0.6. The start point is offset by normal * 0.05 to avoid self-intersection. The result is blended using a Schlick Fresnel term F = (1 − cosθ)⁵ — stronger at grazing angles — via mix(reflectivity * 0.4, reflectivity, fresnel).

5 · Animated Water with Procedural Wave Normals

We model the water surface as a height field h(x,z,t) — a sum of two traveling sine waves with different directions, wavelengths (4.0 and 2.5), speeds (1.5 and 2.1), and amplitudes (0.04 and 0.025). The surface normal is derived via forward finite differences with ε = 0.05: n = normalize(−∂h/∂x, 1, −∂h/∂z). The wave field is evaluated in true world space — reconstructed in gbuffers_water.vsh as (gbufferModelViewInverse * viewVertex).xyz + cameraPosition — so ripples don't drift as the player moves. An isWater flag (set from mc_Entity.x == 102) gates the wave computation away from stained glass, which shares the gbuffers_water stage.

02 — Problems & Lessons

Problems Encountered

ProblemHow We Solved It
Distinguishing block materialsVanilla geometry carries no material identity. We used Iris's block.properties + mc_Entity.x to declare integer material class IDs readable in the vertex shader.
View-space Z sign (SSR)OpenGL view space goes down −Z, so "ray behind geometry" means rayPos.z < scenePos.z. Our first implementation had this backwards, producing wrong-facing reflections. Fixed after tracing through the math on paper.
SSR self-intersection noiseRay starting at exactly viewPos immediately hit the surface it launched from. Fixed by offsetting the ray origin by normal * 0.05.
Stained glass rippling like waterAll translucents share gbuffers_water. We added an isWater varying gated on mc_Entity.x == 102 to restrict wave normals to actual water.
Wave ripples sliding with the cameraIris stores positions camera-relative. Adding cameraPosition to the view-inverse position anchored the wave field to world space.
Shadow acne on flat surfacesA constant depth bias of 0.005 in NDC units was sufficient to eliminate quantization artefacts on all tested surfaces.

Lessons Learned

Deferred shading is a general scaffolding

The geometry-pass → screen-space-pass pattern hosts shadows, SSR, and water normals from the same G-buffer. SSAO, bloom, and volumetrics would all fit in the same framework.

Screen-space techniques are a pragmatic trade-off

SSR is fast and convincing on-screen but cannot reflect off-camera geometry. Knowing and accepting that limit is part of real-time graphics craft.

Height fields → normals via the gradient

The derivation n = normalize(−∂h/∂x, 1, −∂h/∂z) comes from the cross product of surface tangents. Working through it from first principles made the math genuinely ours.

Coordinate frame discipline is everything

Most debugging was not algorithmic — it was tracking which frame (object, world, view, clip, NDC) each vector lived in at each pipeline stage.
